# e2e-test — 사용자 플로우 E2E 테스트 작성 (test-after, 사람 게이트)

이번 work 가 **완성시킨 사용자 플로우**의 E2E 테스트 *코드를 작성*한다. 기기/시뮬레이터가 필요하므로 **자동 실행하지 않고** 사람 게이트로 넘긴다.

## 강제 법칙

> **완성된 플로우만, 사용자 관점으로 짠다. 자동 실행하지 않는다 — 사람 게이트.**

E2E 는 사용자 여정 단위라 단일 work 와 1:1 이 아니다. 플로우가 닫힌 work 에서만 작성한다.

## 언제

- `work` 6.5 에서 이번 work 가 **사용자 플로우를 완성시켰다**고 판단해 호출할 때만.
- 플로우 미완성이면 호출되지 않는다(work 가 "e2e-test: 생략 — 사유"로 기록).

## 책임 경계

| 한다 | 안 한다 |
|---|---|
| E2E 테스트 **코드 + 실행 명령 작성** | 테스트 **자동 실행** (기기 필요 → 사람 게이트) |
| run 파일에 "수동 실행 대기" + 명령 기록 | 모듈 경계 통합 테스트 (→ `integration-test`) |
| 그린필드 가드(규칙 비면 분기) | 필요 여부 **판단** (→ `work` 6.5) |

## 절차

1. **도구 로드**: `@docs/rules/TESTING.md` 에서 E2E 도구/명령(예: Android Espresso/Maestro, iOS XCUITest/Maestro)을 읽는다(단일 출처).
   - ⚠️ **그린필드 가드**: E2E 도구·규칙이 비어 있으면 → 작성을 멈추고 `/harness-root`(또는 `/harness-root-edit testing`) 로 분기해 스택을 먼저 확정한다.

2. **시나리오 설계**: 완성된 사용자 여정을 단계로 쪼개고, 단계마다 사용자가 보는 상태를 단언으로 잡는다.

3. **작성**: TESTING.md 패턴으로 E2E 코드를 쓰고, 실행 명령을 준비한다.

4. **자동 실행 금지**: 돌리지 않는다. `.harness/runs/run-{id}.md` 의 **E2E 수동 실행 대기 목록**에 작성 파일 + 실행 명령을 기록하고 사람 게이트로 넘긴다.

## 시나리오 예시 (무엇이 "사용자 관점 단언"인가)

로그인 플로우를 완성시킨 work 를 예로 든다. 핵심은 사용자가 *실제로 보고 도달하는 상태*를 단언하는 것이다.

**약한 단언:**
```
// 화면이 떴는지만 본다 — 사용자가 로그인에 성공했는지는 모른다
onView(withId(R.id.login_screen)).check(matches(isDisplayed()))
```

**사용자 관점 단언:**
```
// 사용자 여정대로: 입력 → 제출 → 도달한 결과 화면을 단언
onView(withId(R.id.email)).perform(typeText("a@b.com"))
onView(withId(R.id.password)).perform(typeText("pw1234"))
onView(withId(R.id.submit)).perform(click())

// 사용자가 로그인 성공으로 "도달하는" 홈 화면의 환영 문구를 단언
onView(withText("환영합니다, a@b.com")).check(matches(isDisplayed()))
```

## 가드

- 미완성 플로우에 E2E 를 짜지 않는다(work 6.5 가 완성 판단 후에만 호출하지만, 스킬 안에서도 재확인한다).
- "화면이 떴다"가 아니라 사용자 관점의 결과 상태를 단언한다.

## 원칙

- 스킬=작성기. 실행은 기기 위에서 사람이 트리거한다(자동 루프 비편입).
- 규칙은 `TESTING.md` 단일 출처. 도구/명령을 여기 하드코딩하지 않는다.
