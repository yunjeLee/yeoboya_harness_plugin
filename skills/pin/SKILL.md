# pin — 현재 상태 스냅샷

진행 중인 work 의 현재 상태를 활성 run 파일에 박아둔다. 세션이 끊겨도 다음 세션이 이어받을 수 있도록, 지금까지의 진행을 영속화하는 것이 목적이다.

work 는 자기 흐름의 5단계에서 run 파일을 갱신하지만, 그 사이 중간 점검이나 세션 종료 직전에 **수동으로** 현재 상태를 고정하고 싶을 때가 있다. pin 이 그 역할을 한다.

## 트리거
- `/pin` — 현재 상태를 자동 파악해 갱신
- `/pin <메모>` — 자동 파악 + 메모를 결정 로그에 추가

## 흐름

### 1. 활성 run 파일 탐색
`.harness/runs/run-*.md` 중 **미완료**(남은 완료기준 체크리스트에 `- [ ]` 가 하나라도 남은) run 을 찾는다. work 0.5 재개 점검과 동일한 기준이다.

- **없음** → "활성 작업이 없습니다. `/work` 로 작업을 시작하세요." 안내 후 **중단**한다. pin 은 *갱신* 도구지 *생성* 도구가 아니다. run 파일을 새로 만들지 않는다.
- **여러 개** → 목록(요구사항 한 줄 요약 + 현재 단계)을 보여주고 어느 작업을 갱신할지 사람에게 묻는다.
- **하나** → 그 파일을 대상으로 다음 단계로 진행한다.

### 2. 자동 파악 (현재 대화 맥락 기준)
지금 세션에서 실제로 진행된 내용을 근거로 run 파일의 해당 섹션을 갱신한다. **추측하지 말 것** — 대화에서 확인되는 사실만 반영한다.

- **현재 단계**: work 흐름 중 지금 어디인지 (예: "6. TDD 구현 중", "7. 완료기준 검증 대기").
- **완료한 완료기준**: 검증으로 통과가 확인된 항목만 `- [ ]` → `- [x]` 로 바꾼다. 아직 안 끝난 건 그대로 둔다.
- **bug-fix 시도 횟수**: 이번 세션에서 bug-fix 가 돌았으면 완료기준별 누적 횟수를 갱신한다.
- **남은 완료기준 체크리스트**: 위 반영 결과로 자연스럽게 갱신된다.

근거가 불확실한 항목은 건드리지 않는다. 잘못된 상태를 박아두면 다음 세션이 그걸 사실로 믿고 이어받아 더 큰 혼란이 된다 — 그래서 "확인된 것만" 이 중요하다.

### 3. 수동 병합 (메모가 있을 때)
`/pin <메모>` 로 텍스트가 붙으면 run 파일의 **결정 로그**에 한 줄 추가한다.

```
- {KST} <메모>
```

KST 시각은 `TZ=Asia/Seoul date +%Y%m%d-%H%M` 로 얻는다. 결정 로그는 누적이므로 기존 줄을 지우지 말고 **append** 한다.

### 4. 결과 요약
무엇이 바뀌었는지 사람에게 한눈에 보여준다.

```
📌 pin: run-{id}.md 갱신
- 현재 단계: <이전> → <이후>
- 완료기준: <체크된 항목 수>/<전체>
- 결정 로그: +<추가된 줄 수>줄
```

## 경계 (하지 않는 것)
- **갱신만** 한다. 완료기준 검증·bug-fix·분기는 work 와 해당 스킬의 몫이다. pin 은 상태를 *기록*할 뿐 작업을 *진행*하지 않는다.
- run 파일을 **새로 만들지 않는다**. 생성은 work 의 책임이다.
- 확인되지 않은 진행 상태를 추측해 채우지 않는다.

## 원칙
- run 파일 골격(work 스킬 정의: 요구/완료기준/현재 단계/bug-fix 횟수/남은 완료기준 체크리스트/결정 로그)을 **단일 출처**로 사용한다. pin 이 임의 섹션을 새로 만들지 않는다.
- 핸드오프는 파일 경유 — pin 이 박아둔 상태를 다음 세션의 work 가 그대로 읽어 이어받는다.
