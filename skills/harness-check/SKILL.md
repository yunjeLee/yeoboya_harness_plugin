# harness-check — 하네스 문서 문제 진단/기록

산출물이 하네스 기준과 다르거나, bug-fix 루프가 막혔을 때 "하네스가 잘못 가이드했나"를 진단한다. **자동 수정하지 않는다** — 기록만 (하네스=규칙 변경은 사람 게이트).

## 트리거
- `/harness-check {불만 사항}` — 사람이 직접 호출
- **자동 호출**: bug-fix 루프 5회 초과로 중단됐을 때

## 절차
1. **불만 수집**: 인자로 어떤 산출물이 하네스 문서의 무엇과 다른지 받는다. 없으면 묻는다.

2. **문서 통독**: 관련 하네스 문서(루트 + 해당 모듈 CLAUDE.md)를 모두 읽는다.

2.5. **원인 분류 (triage) — 문서 수정 권고 전 필수**: 이 실패가 정말 문서 문제인지 먼저 가린다.
   ① **문서 오류**   — 문서가 실제로 틀림/누락. → 3번으로 (권고 수정 기록).
   ② **문서 미준수** — 문서는 정확한데 LLM 이 안 따름. → 문서 안 건드림. run 파일에 "문서 정확/미준수" 기록 후 work 로 1회 재시도 회수.
   ③ **작업 난이도** — 문서 무관, 본질적으로 어려움. → 문서 안 건드림. 사람 에스컬레이션(작업 분해 권고).
   ④ **환경/의존성** — 에뮬레이터·의존성·인증 등. → 문서 안 건드림. 환경 점검 항목을 사람에게 보고.
   - **①일 때만** 아래 3·4번(문서 권고·기록)으로 진입한다. ②③④는 여기서 종료하고 목적지로 보낸다.

3. **진단 기록 (triage ①일 때만)**: 아래 항목으로 정리한다.
   - 불만 사항 / 대상 모듈 / 관련 문서 / 관련 내용 및 설명 / 권고 수정 내용

4. **Notion 기록**: Notion 에 위 내용을 페이지로 기록한다.
   - **실패 시 로컬 폴백**: 인증/네트워크 문제로 실패하면 `docs/harness-issues/{날짜}-{slug}.md` 에 기록하고 루프를 멈추지 않는다.
   
5. **사람 게이트**: 실제 문서 수정은 사람이 검토 후 harness-*-edit 로 진행. 절대 자동 수정 금지.

## 닫힌 루프 연결
- 코드 루프(work/bug-fix)가 막히면 여기서 하네스 개선 루프로 넘어가 전체가 닫힌다.
