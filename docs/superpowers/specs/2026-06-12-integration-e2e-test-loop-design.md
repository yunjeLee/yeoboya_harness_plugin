# 통합/E2E 테스트를 닫힌 루프에 통합 — 설계 문서

작성일: 2026-06-12

## 1. 문제

현재 plugin 은 `superpowers:tdd` 로 unit 테스트(test-first)를 작성하지만,
AI 자동화 루프가 신뢰성을 가지려면 **모듈 경계를 넘는 통합/E2E 테스트**가 필요하다.
그런데 통합/E2E 테스트 *코드를 작성*하는 워크플로우가 없다.

핵심 구분:
- **검증층** (완료기준 → 명령 실행 → bug-fix): `TESTING.md` 에 이미
  build/lint/Unit/Integration/E2E 슬롯이 있어 통합테스트를 *돌리는* 건 가능.
- **작성층** (테스트 코드를 누가 쓰나): `superpowers:tdd` 만 존재 → unit 중심.
  통합/E2E 테스트 *코드 작성* 단계가 비어 있다.

TDD 가 책임지는 경계: `superpowers:tdd` 는 "테스트 레벨"을 정의하지 않는다.
범위 기준은 **"새로 짜는 함수/메서드/클래스 하나당 테스트 하나"** 이며,
mock 을 지양하고 real code 를 권장하므로 *한 모듈 안* 의 sociable unit test 까지는
자연스럽게 커버된다. 비어 있는 건 그 **밖** — 모듈↔모듈 경계와 기기 E2E 뿐이다.

## 2. 결정 요약

| 항목 | 결정 |
|---|---|
| 방법론 | unit = TDD(test-first), 통합/E2E = **test-after** |
| 실행 환경 | JVM(unit + Robolectric/in-memory 통합) = 자동 bug-fix 루프 / E2E(기기) = 사람 게이트 |
| 구조 | `work` 에 단계 삽입(새 스킬·트리거 없음), 규칙은 `TESTING.md` 단일 출처 |
| 작성 범위 | 이번 변경이 **가로지른 모듈 경계 자동** + 완료기준 명시 시 확장 |
| 실행 범위 | 전체 스위트(회귀 포함) |

방법론 근거: Android/실무 표준은 "unit=TDD, 통합/E2E=test-after"(테스트 피라미드).
outside-in TDD 는 E2E red-green 마다 에뮬레이터가 필요해 비용이 커 채택하지 않는다.

구조 근거(방향 A): 빠진 건 "작성 단계" 하나뿐이라 단계 삽입이 최소 변경이고,
test-after 는 TDD 같은 엄격한 절차가 없어 별도 스킬로 뗄 무게가 아니다(과분리 회피).
`superpowers:tdd` 는 외부 의존이라 확장 불가하므로 그 스킬을 손대지 않는다.

## 3. 작성 범위 vs 실행 범위 (핵심)

`work` 는 "이번 작업 단위"로 도는 닫힌 루프다. 따라서:

| | 작성 범위 | 실행 범위 |
|---|---|---|
| 통합(JVM) | 이번 변경이 가로지른 모듈 경계 | 전체 스위트 (회귀 포함) → 자동 bug-fix |
| E2E(기기) | 이번 work 가 완성시킨 플로우만 | 전체 / 사람 게이트 |

- **작성**을 변경분으로 한정해야 매번 전체 스위트를 다시 쓰지 않고 루프가 닫힌다.
- 안 건드린 경계의 회귀는 **실행 = 전체**로 잡는다. 통합 커버리지는 work 누적으로 쌓인다.
- E2E 는 사용자 플로우 단위라 단일 work 와 1:1 이 아니다 → 플로우가 닫히는 work 에서만 작성.

## 4. 부트스트랩 (TESTING.md 가 비어 있을 때)

TESTING.md 는 "테스트 코드 요약"이 아니라 **"테스트 작성 규칙(라이브러리/명령/네이밍)"** 이다.
테스트 코드가 0개여도 규칙은 존재할 수 있고, `harness-root` 가 하네스 생성 시 채운다.

| 시점 | TESTING.md 상태 | 6.5 동작 |
|---|---|---|
| 하네스 생성 후 (정상) | 스택·명령 결정 있음 (코드 0개여도) | 규칙 읽고 통합/E2E 작성 |
| 그린필드 (스택 미정) | 비어 있음 | **멈춤 → `/harness`로 분기**해 스택 먼저 결정 |
| work 누적 후 | 실제 테스트 패턴까지 구체화 | 기존 패턴 따라 작성 |

그린필드 가드는 기존 패턴과 일치한다(bug-fix 5회 → harness-check 처럼,
루프가 막히면 하네스 개선 루프로 넘어가 전체가 닫힌다).

## 5. 변경 대상

### 파일 1: `skills/work/SKILL.md`
- ① 2번(완료 기준 번역)에 부트스트랩 가드: TESTING.md 통합/E2E 규칙·명령이 비면 `/harness` 분기.
- ② 6번과 7번 사이 `6.5` 신설: test-after 통합/E2E 작성(작성 범위·레벨 분기·E2E 사람 게이트·단언 가드).
- ③ 7번에 "전체 JVM 스위트 실행"(회귀 검출) 명시.
- ④ 상태 파일 골격에 "E2E 수동 실행 대기 목록" 추가.

### 파일 2: `shared/templates/root-docs.md` (TESTING.md 스키마)
- ⑤ 통합/E2E 작성 규칙(경계 동작 단언 가드) + 실행 명령 JVM/기기 분리 표기.

### 파일 3: `skills/bug-fix/SKILL.md`
- 변경 없음. "완료기준 명령 실패 → 수정 → 재검증" 구조가 통합테스트 실패도 그대로 처리(기존 루프 재사용).

## 6. 한 줄 요약

> work(6.5 신설) + TESTING.md(작성 규칙·실행 분기) 만 손대고 bug-fix 는 그대로 재사용한다.
> TDD 가 못 채우는 "모듈 경계 밖"을, test-after 작성(변경분) + 전체 실행(회귀)으로
> 닫힌 루프 안에서 메운다.
