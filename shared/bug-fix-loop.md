# bug-fix 루프 — 검증 실패 자동 수정

work 의 완료기준 검증이 실패하면 work 가 이 절차를 Read 해 자기 컨텍스트에서 실행한다.
**외부에서 직접 호출하지 않는다 — work 내부에서만 진입한다.** 에러코드 추출 → 자동 수정 → 재검증을 반복한다.

## 진입 조건
- work 7단계에서 `completion-verifier` 결과가 `has-failure` 일 때만 진입.
- 입력: 실패한 완료기준 명령 + `.harness/logs/{명령slug}.log` + run-id.

## 절차
1. **에러 추출**: work 가 남긴 `.harness/logs/{명령slug}.log` 의 **마지막 실패 블록**에서 에러코드·메시지·파일:라인을 추출한다.
   - 빌드 도구별 인지 힌트: Gradle → `> Task :모듈:test FAILED`, `e: file:line:`, `FAILURE:` / xcodebuild → `error:`, `** TEST FAILED **`.
   - 추출한 에러의 **원인 시그니처**(에러코드)를 `.harness/runs/run-{id}.md` 에 기록한다. (3번 동일 원인 재발 판정에 사용 — **에러코드만**으로 비교한다. 파일:라인/메시지는 판정에 쓰지 않는다.)

2. **자동 수정**: 근본 원인을 좁혀 코드를 수정한다. (임시 회피 금지 — 구조적 해결)
   - **결정 로그 (필수)**: 재검증 직후 `.harness/runs/run-{id}.md` 의 "결정 로그"에 1줄 append 한다.
     형식: `시도{N} [{에러코드}] → 가설:{한 줄} → 수정:{파일:심볼} → 결과:{통과|실패(원인)}`
     (사람 개입 시 맥락 복원·사후 감사용. raw 로그가 아니라 요약 서사다.)

3. **재검증**: 실패한 완료기준 명령의 재실행을 `completion-verifier` 에이전트에 위임한다. 직접 실행하지 않고 결과(통과/실패 + 원인 시그니처)만 받아 분기한다 (무거운 테스트 출력의 메인 컨텍스트 오염 방지 — work 7단계와 동일 원칙).
   - **위임 입력**: 직전에 실패한 단일 완료기준 명령 + run-id (로그는 `.harness/logs/{명령slug}.log` 에 갱신됨).
   - 통과 → **이 명령의 카운터를 종료**하고 다음 완료기준으로 / 모두 통과면 작업 완료.
   - 실패(다른 원인) → **이 명령의 시도 횟수 +1** 후 1번으로.
   - 실패(**직전과 동일한 원인 시그니처**) → 수정이 같은 원인을 못 잡은 것이므로 **검증을 즉시 중단**하고 `/harness-check` **자동 트리거**(5회 대기 없이). "같은 원인 A 가 수정 후에도 재발 = 코드 단위로 해결 불가, 하네스 가이드 문제일 가능성" 진단으로 넘긴다.

4. **카운터**: 시도 횟수는 **완료기준 명령별로 독립**이며 `.harness/runs/run-{id}.md` 에 명령별로 기록한다. 한 명령당 **최대 5회**. 세션 재개 시 이 파일에서 명령별 횟수를 읽어 이어간다.

5. **harness-check 분기 (둘 중 먼저 걸리는 쪽)**: 루프 **중단** → `/harness-check` **자동 트리거** ("이 실패가 하네스 문서의 잘못된 가이드 때문인가?" 진단 → 로컬 기록). 코드 루프가 막히면 하네스 개선 루프로 넘어가 전체가 닫힌다.
   - (a) 한 명령의 시도 횟수 **5회 초과**.
   - (b) 수정 후 재검증에서 **직전과 동일한 원인 시그니처** 재발(3번) — 5회를 기다리지 않고 즉시.

## 원칙
- 검증·카운터는 work 컨텍스트 안에서 이 절차로 처리한다.
- 상태 파일로 시도 횟수를 영속화해 세션이 끊겨도 카운트가 유지된다.
